The compaction tool under `/app/environment` is producing evidence that can look healthy for a clean run but does not hold up across restart, rollback, and idempotent rerun workflows. Fix the source code under `/app/environment`, not just the generated output, so the normal runner command `go run /app/environment/e5 --write-report /app/output/ledger_compaction_report.json` regenerates `/app/output/ledger_compaction_report.json` from the bundled local histories; the verifier reruns that command.

The JSON report uses schema_version 1 and carries command, runs, artifacts, and state_transitions. Runs are labeled clean, restart, rollback, and idempotent. Their records and provenance carry id, generation, value, and source observations; transitions carry label, record_count, and max_generation; artifacts carry path and row_count, where each row_count is the number of rows in that bundled input file. The bundled active record identifiers are A100, B200, and C300; D400 is a retired input row that must not appear as active output. When two candidates share the same identifier and generation, keep the row whose source has higher rank in the rank table bundled with the primary histories. The state_digest is built by sorting active records by id, formatting each as id:generation:value:source, and joining those parts with |. Consecutive regenerations of the report must yield the same state_digest and matching record and provenance observations per run label.

The bundled histories are input data, not outputs to rewrite. Static or manual output writes are insufficient, and test edits, wrapper-only changes, or bypassing the Go pipeline are not acceptable; the normal pipeline must regenerate the report after stale output is deleted and the workflows are rerun.
