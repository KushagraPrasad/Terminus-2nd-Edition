## instruction.md
A local offline Go module proxy has been deployed in the container environment. The proxy implementation sources are located under `/app/server`. However, developers are reporting inconsistent behavior when interacting with the proxy. First, running `go list -m -versions <module>` reports one set of active versions. Second, running `go get <module>@latest` resolves to a completely different version that is not reported in the list. Third, certain historical versions can still be downloaded directly if queried explicitly, despite being absent from the list. Fourth, fresh workspaces (or checkouts with clean module caches using `go clean -modcache`) fail to resolve or build, whereas developers with pre-existing, cached builds can compile without issues. Your task is to identify the synchronization and metadata caching issues causing this divergent behavior and repair the proxy implementation. Below are the requirements for this task.
1. Repair the proxy source code under `/app/server`.
2. Compile the repaired binary to verify your changes.
3. Ensure that all endpoints of the local proxy serve a consistent view of the module repository.
4. Clean workspace checkouts must successfully resolve and download all active versions.
5. Deleted/pruned versions must not be served or cached.
6. The storage files on disk are saved in the directory structure under `/app/data/storage/github.com/user/repo/@v`.
7. The verifier will generate a test report JSON at `/app/output/verifier_report.json` containing the status of the following checks.
   - `test_gc_purges_storage_files` checks that purging a version via GC deletes the corresponding files from storage.
   - `test_gc_invalidates_latest_cache` checks that purging a version updates/invalidates the latest metadata cache.
   - `test_latest_resolves_active_version` checks that the latest endpoint resolves to the correct active version.
   - `test_latest_meta_cache_invalidation` checks that the latest endpoint does not serve stale cached metadata.
   - `test_list_endpoint_filtering` checks that the list endpoint only returns versions that exist in the database and on the filesystem.
   - `test_fresh_workspace_resolution` checks that the proxy behaves consistently for fresh workspace builds and lists.
Please make sure that the server operates correctly under all circumstances. To solve this task, you must repair the Go source files under `/app/server`. The verifier's automated tests interact with the local proxy, using standard Python library components such as `socket` for network communication, `sqlite3` to check database states directly, and `urllib` to send HTTP requests. Static or manual output writes are insufficient; you must fix the proxy implementation so that the normal test pipeline successfully executes the tests and regenerates the report at `/app/output/verifier_report.json`.
