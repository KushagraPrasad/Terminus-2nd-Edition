# Incremental Build Graph Convergence Failure

Fix the Go build engine under /app/environment so repeated build runs converge instead of leaving stale graph, fingerprint, registry, or filesystem state behind. The binary is build_engine, and the verifier rebuilds it from the submitted source before exercising the build workflow. Static or manual output writes are insufficient; the implementation has to update the SQLite database, output directory, and manifest from the current action graph each time it runs.

The engine reads a dependency graph from /app/action_graph.json by default, stores state in /app/build.db using sqlite3-compatible SQLite access, and writes the active artifact manifest to /app/outputs/artifact_manifest.json. The action_graph JSON contains dependency edges and actions; database reads may fetchone or fetchall rows from tables such as artifacts, where file_path is the artifact key, and a target is the active artifact path or action identifier being built. Here action_graph means the JSON build graph, actions means the build steps inside that graph, fetchone and fetchall mean reading one or all SQLite result rows, file_path means the artifact table key, and artifact_manifest means the JSON manifest of active artifacts. Dependency edges in the database must exactly match the current graph after each build: removed edges disappear, new edges are inserted, an empty graph leaves 0 active edges, and cyclic graphs are rejected instead of being persisted as a valid build. This needs to work through the library behavior as well as the CLI.

Action fingerprints must be stable for changes that do not affect a build contract and sensitive to changes that do. For Go inputs, only exported interface signatures should influence the fingerprint, so comments, whitespace, and private helper implementation changes should not force downstream rebuilds. For non-Go inputs, the fingerprint must be derived from the raw file contents. Registry synchronization must write a JSON object from active artifact paths to lowercase SHA-256 fingerprint hash strings of length 64 using the 0123456789abcdef alphabet and remove physical files in the outputs directory that no longer correspond to active database artifacts, so their file-existence checks are false. Invoking the binary without a command must print a usage message prefixed with Usage: for build_engine to standard error. The observable contract includes dependency edge reconciliation, cycle rejection, Go interface fingerprint stability, non-Go raw content fingerprinting, manifest fingerprint synchronization, and stale output pruning.
