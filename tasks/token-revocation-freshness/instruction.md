# Token Revocation and Freshness Coordinator
A distributed token authorization gateway is accepting stale authority decisions after token revocations, key refresh failures, clock-skewed validation, and blacklist synchronization events. Fix the gateway implementation under `/app/environment` so the source logic, not a hand-written output file, produces correct authorization traces when the verifier runs the scenario processor. The gateway validates JWTs with a JWKS-backed verifier, a short-lived in-memory authorization cache, and a local SQLite blacklist store. Revocation state is delivered as ordered synchronization batches. A correct fix should make these layers agree immediately: (1) Cached decisions must not outlive exact or wildcard revocations. (2) Wildcard invalidation patterns (e.g., principal:user-100) are formatted as field:value, where the prefix dictates the cache entry field to match. (3) Blacklist updates must be applied atomically and only when their sequence advances the current epoch. (4) Token expiration must be evaluated against the request skew offset when one is supplied. (5) Cached JWKS material must fail closed when a refresh timeout leaves only expired fallback keys. The verifier drives the compiled TypeScript scenario runner with generated operation files using `node /app/environment/dist/scenario.js scenario.json` and inspects the audit trace at `/app/logs/verifier/auth_audit_trace.json`. Static or manual writes to that trace are insufficient; the normal runner must regenerate it. The trace JSON contains a results array whose entries expose status, result, revoked, and sequence fields. Failed operations report a status of error. Token validation results report valid as true only for accepted authorizations and false for revoked, expired, or failed checks, while database revocation probes report revoked as true only for committed matching revocations and false after rejected or rolled-back writes. When a token is rejected because it is revoked, the result includes the exact revocation reason Token revoked. When a token is rejected due to an expired fallback key, the reason string must contain the word expire. The verifier checks behavioral outcomes for cache wildcard invalidation, clock skew offset verification, jwks key fallback on network timeout, database transaction crash persistence, and monotonic epoch sequence validation, using representative timing and sequence values including 100, 200, 1000, 11111, and 9999999999. The scenario configuration schema must define a list of operations to execute sequentially. These inputs may include dbPath, keyCacheSeed, operations, expiresAt, skewMs, simulateTimeout, type, token, batch, sequence, revocations, id, principal, scope, timestamp, pattern, invalidateByPattern, checkDbRevoked, and getHighestSequence. The local database path may be gateway.db. Generated JWT payloads and key labels may use representative values such as user-123, user-100, user-200, user-first, user-other-, jti-1, jti-2, jti-a, jti-b, jti-first, jti-other-, kid-rot-1, kid-exp-1, kid-wild-1, principal:user-100, rev-id-1, rev-ok-1, rev-seq-5, rev-seq-10, and a trigger-crash-sig record for rollback behavior. JWKS verification and fallback cache seed data may include mock-public-key-for-kid-k, mock-public-key-for-kid-kid-rot-1, mock-public-key-for-kid-kid-exp-1, and mock-public-key-for-kid-kid-wild-1. The in-memory authorization cache has capacity 1000 and should evict the oldest entries when over capacity. The exact scenario identities are intentionally not part of this prompt; implement the general gateway behavior in the existing cache, store, JWKS, validator, and scenario code. Run `npm test` from `/app/environment` after rebuilding to exercise the same integration path used by the verifier.
