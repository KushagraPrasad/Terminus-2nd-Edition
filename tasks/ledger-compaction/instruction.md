The compaction tool under `/app/environment` is producing evidence that can look healthy for a clean run but does not hold up across restart, rollback, and idempotent rerun workflows. Fix the source code under `/app/environment`, not just the generated output, so the normal runner command `go run /app/environment/e5 --write-report /app/output/ledger_compaction_report.json` regenerates `/app/output/ledger_compaction_report.json` from the bundled local histories; the verifier reruns that command.

The JSON report uses schema_version 1 and carries command, runs, artifacts, and state_transitions. The runs field is a JSON array of objects, each with a label field set to clean, restart, rollback, or idempotent; do not emit runs as a map keyed by label. Each run object carries records, provenance, state_digest, and transitions. Records and provenance both list one entry per active record; provenance must mirror records with the same id set and matching id, generation, value, and source on each entry. Transitions carry label, record_count, and max_generation, where record_count equals the number of active records in that run. The top-level state_transitions array aggregates one transition object from each run label. The artifacts array lists exactly these four bundled row-source files and no others: `/app/environment/f6/r1.json`, `/app/environment/f6/r2.json`, `/app/environment/g8/r1.json`, and `/app/environment/g8/r2.json`. Each artifact entry carries path and row_count, where row_count is the number of rows in that file. Do not include the rank table or other auxiliary files in artifacts. The report must not contain verdict sentinel substrings such as `_ok`, `_green`, `_valid`, `_passes`, or `stays_green`.

For the clean run, merge rows from `/app/environment/f6/r1.json` and `/app/environment/f6/r2.json` using the rank table bundled with the primary histories. Bundled row sources use the names anchor, cache, and audit; the rank table assigns each source name a numeric rank for same-generation tie-breaking. Restart and idempotent runs additionally incorporate rows from `/app/environment/g8/r1.json`. Rollback and idempotent runs additionally incorporate rows from `/app/environment/g8/r2.json`. Active rows contributed only through rollback workflow inputs must appear on rollback and idempotent runs and must not appear on clean or restart runs. The bundled active record identifiers are A100, B200, and C300; D400 is a retired input row that must not appear as active output. When two candidates share the same identifier and generation, keep the row whose source has higher rank in the rank table bundled with the primary histories. The state_digest is built by sorting active records by id, formatting each as id:generation:value:source, and joining those parts with |. Consecutive regenerations of the report must yield the same state_digest and matching record and provenance observations per run label.

The bundled histories are input data, not outputs to rewrite. Static or manual output writes are insufficient, and test edits, wrapper-only changes, or bypassing the Go pipeline are not acceptable; the normal pipeline must regenerate the report after stale output is deleted and the workflows are rerun.
